# Refined Request: Gitter Web UI

## 1. Summary

Add a new `gitter ui` command that launches a lightweight local web server (e.g., `http://localhost:3000`) and opens the default browser. The web page presents a rich, interactive view of the gitter registry, allowing the user to browse, filter, sort, and inspect repository entries with rendered markdown descriptions and notes.

---

## 2. New Command

**Command:** `gitter ui`

**Options:**

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `--port <number>` | number | 3000 | Port for the local web server |
| `--no-open` | boolean flag | (opens by default) | Suppress automatic browser launch |

**Behavior:**

- Starts an HTTP server on `localhost:<port>`.
- Opens the default browser to `http://localhost:<port>` (unless `--no-open` is specified).
- Serves a single-page HTML application with all CSS and JS inlined (no external frontend dependencies, no build step).
- Exposes a JSON API endpoint (`GET /api/registry`) that returns the current registry data from `~/.gitter/registry.json`.
- The server runs until the process is interrupted with `Ctrl+C` (SIGINT/SIGTERM), at which point it shuts down gracefully.
- Prints the server URL to stderr on startup (consistent with gitter's stdout/stderr discipline).
- If the requested port is already in use, the command must fail with a clear error message (no silent fallback to another port).

---

## 3. Frontend Specification

### 3.1 Technology Constraints

- **No external frontend dependencies.** Everything (HTML, CSS, JS) must be inlined in a single HTML string or file served by the Node.js server.
- **No build step.** No React, Vue, webpack, Vite, or similar. Vanilla HTML/CSS/JS only.
- **Markdown rendering:** Use a lightweight, dependency-free markdown-to-HTML converter bundled inline (e.g., a minimal marked.js or similar library embedded in the HTML, or a custom lightweight parser). The converter must handle the common markdown features used in gitter descriptions: headings, bold, italic, lists, code blocks, inline code, links, and paragraphs.

### 3.2 Page Layout

The UI is a **single-page application** with two panels:

```
+--------------------------------------------------+
|  HEADER: "Gitter Registry" + repo count + filter  |
+---------------------+----------------------------+
|                     |                            |
|  REPO LIST          |  DETAIL VIEW               |
|  (left panel,       |  (right panel,             |
|   scrollable)       |   scrollable)              |
|                     |                            |
|  [repo card]        |  Repo metadata             |
|  [repo card]        |  Rendered description      |
|  [repo card]        |  Rendered notes            |
|  ...                |                            |
|                     |                            |
+---------------------+----------------------------+
```

- **Responsive:** On narrow screens (< 768px), the panels stack vertically (list on top, detail below).
- **Visual style:** Clean, modern, dark or light theme (developer-friendly). Monospace font for paths and branch names. Sans-serif for descriptions and UI text.

### 3.3 Repository List Panel (Left)

Each entry in the list displays:

- **Repository name** (bold, primary text)
- **Local path** (truncated, monospace, secondary text)
- **Current branch** (badge/tag style)
- **Status indicators:** icons or badges showing whether the entry has a description and/or notes
- **Last updated** (relative time, e.g., "3 days ago")

Clicking an entry selects it and populates the detail panel on the right. The selected entry is visually highlighted.

### 3.4 Detail View Panel (Right)

When an entry is selected, the detail panel shows:

#### Metadata Section
- **Repository name** (heading)
- **Local path** (full, monospace, copyable)
- **Current branch**
- **Local branches** (list or comma-separated)
- **Remote branches** (list or comma-separated, collapsible if many)
- **Remotes** (table: name, fetch URL, push URL)
- **Last updated** (absolute ISO timestamp + relative)

#### Description Section (if present)
- **Business Description** heading, followed by the `businessDescription` markdown rendered to HTML
- **Technical Description** heading, followed by the `technicalDescription` markdown rendered to HTML
- **Generation metadata:** generated at (timestamp), generated by (model), instructions used (if any)
- If no description exists, show a muted placeholder: "No description available. Run `gitter describe` to generate one."

#### Notes Section (if present)
- **Notes** heading, followed by the `notes` markdown rendered to HTML
- If no notes exist, show a muted placeholder: "No notes. Run `gitter notes` to add some."

### 3.5 Filtering and Sorting

#### Filter Bar (in the header area)

| Filter | Type | Behavior |
|--------|------|----------|
| **Text search** | text input | Filters by repo name, local path, and remote URLs (case-insensitive partial match) |
| **Has description** | toggle/checkbox | Show only repos that have a description |
| **Missing description** | toggle/checkbox | Show only repos that lack a description |
| **Has notes** | toggle/checkbox | Show only repos that have notes |
| **Missing notes** | toggle/checkbox | Show only repos that lack notes |

- Filters are combined with AND logic.
- The list updates immediately as filters change (no submit button).
- The repo count in the header updates to reflect the filtered count (e.g., "Showing 5 of 23 repositories").

#### Sorting

| Sort option | Direction |
|-------------|-----------|
| Name | A-Z / Z-A |
| Last updated | Newest first / Oldest first |
| Path | A-Z / Z-A |

- Default sort: Name A-Z.
- Sort control: a dropdown or clickable column headers.

## 4. Backend Specification

### 4.1 Server Technology

- Use Node.js built-in `http` module (no Express, no Koa, no external server dependencies).
- The server module lives at `src/commands/ui.ts` (command handler) and `src/web-server.ts` (server logic).

### 4.2 API Endpoints

| Method | Path | Response | Description |
|--------|------|----------|-------------|
| `GET /` | `text/html` | The full single-page HTML application | Serves the UI |
| `GET /api/registry` | `application/json` | `{ repositories: RegistryEntry[] }` | Returns the current registry data |

### 4.3 Data Flow

1. On `GET /api/registry`, the server calls `loadRegistry()` from `src/registry.ts` to read the latest data from disk.
2. The frontend fetches `/api/registry` on page load and whenever the user wants to refresh.
3. The registry is read fresh on each API call (not cached), so changes made via CLI commands in another terminal are reflected when the page is refreshed.

### 4.4 Graceful Shutdown

- Listen for `SIGINT` and `SIGTERM`.
- Close the HTTP server and exit cleanly.
- Print a shutdown message to stderr.

## 6. Dependencies

### New Dependencies Required

- **`open`** (npm package) -- to open the default browser cross-platform. This is the only new dependency. Alternatively, use `child_process.exec` with platform-specific commands (`open` on macOS) to avoid the dependency entirely, since the project targets macOS.

### No New Frontend Dependencies

All frontend code (HTML, CSS, JS, markdown parser) must be inlined in the served HTML. A minimal markdown parser can be implemented in ~50 lines of JS to handle the subset of markdown used by gitter descriptions (headings, bold, italic, lists, code blocks, inline code, links, paragraphs).

---

## 7. Non-Functional Requirements

- **No persistent state:** The web UI is read-only. It reads the registry but never writes to it.
- **No authentication:** The server binds to localhost only and is not accessible from the network.
- **Performance:** The registry is typically small (tens to low hundreds of entries). No pagination needed.
- **Port conflict:** If the port is in use, fail immediately with an error message. Do not silently pick another port (per project rule: no fallback behavior).
- **Browser opening:** On macOS, use the `open` command via `child_process.exec('open <url>')` to launch the default browser. This avoids adding an npm dependency.

---

## 8. Acceptance Criteria

1. `gitter ui` starts a local web server on port 3000 and opens the browser.
2. `gitter ui --port 8080` uses port 8080 instead.
3. `gitter ui --no-open` starts the server without opening the browser.
4. The web page loads and displays all registered repositories in a list.
5. Clicking a repository shows its full metadata, rendered description, and rendered notes in the detail panel.
6. Text search filters the list by name, path, or remote URL as the user types.
7. Toggle filters for has/missing description and has/missing notes work correctly.
8. Sorting by name, last updated, and path works in both directions.
9. The filtered count updates dynamically in the header.
10. Markdown in descriptions and notes renders correctly (headings, lists, bold, italic, code blocks, inline code, links).
11. `Ctrl+C` in the terminal stops the server gracefully.
12. If port is occupied, the command exits with a clear error.
13. The server binds to `127.0.0.1` only (not `0.0.0.0`).
14. No external frontend resources are loaded (no CDN links, no fetch to external URLs).

---

## 9. Out of Scope

- Editing or writing to the registry from the web UI.
- User authentication or access control.
- WebSocket or live-reload (user manually refreshes to see registry changes).
- Persistent UI preferences (theme, sort order, etc.).
- Mobile-optimized layout beyond basic responsive stacking.
